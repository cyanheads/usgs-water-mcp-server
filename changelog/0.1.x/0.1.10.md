# 0.1.10 — 2026-07-16

## Added

- **`water_get_conditions` `historicalContextStatus`** — a required top-level enum (`available` / `no_matching_day` / `no_record` / `unavailable`) stating why `historicalContext` is or is not populated. `unavailable` marks a failed statistics-service call as transient and retryable, never attributing it to the site's own record. Rendered in `format()`. ([#20](https://github.com/cyanheads/usgs-water-mcp-server/issues/20))
- **`water_get_conditions` `historicalContext.comparisonBasis`** — a fixed disclosure, present whenever `historicalContext` is non-null, that `percentileClass` ranks an instantaneous reading against approved daily-mean percentiles — a cross-granularity approximation, not a flood-stage or drought determination. Carried as its own field because schema description text is invisible wherever the raw value is read; rendered in `format()`. ([#17](https://github.com/cyanheads/usgs-water-mcp-server/issues/17))
- **`water_get_conditions` `invalid_request` error reason** (`ValidationError`) — a well-formed `site` / `parameterCd` that NWIS still rejects now surfaces as this typed reason, carrying NWIS's own message, rather than an opaque upstream error. ([#18](https://github.com/cyanheads/usgs-water-mcp-server/issues/18))

## Changed

- **`water_get_conditions` `site` / `parameterCd` inputs** — now validated with the shared `SiteNumberSchema` / `ParameterCdSchema` (8–15 digit / 5-digit patterns) instead of bare `z.string()`, matching the other NWIS-backed tools; the constraints advertise in the tool's JSON Schema `pattern`, so a malformed value is rejected at the edge instead of consuming a network call. ([#18](https://github.com/cyanheads/usgs-water-mcp-server/issues/18))
- **`water_get_conditions` framing** — dropped the "is this flooding or drought?" line from the `@fileoverview`, tool `description`, `README.md`, and `docs/design.md`; the classification is now stated as a "how unusual is this reading for this calendar day" percentile rank against approved daily-mean statistics, not a flood-stage or drought determination. ([#17](https://github.com/cyanheads/usgs-water-mcp-server/issues/17))

## Fixed

- **`water_get_conditions` stat-failure attribution** — the parallel stat call is captured as a tagged outcome instead of `.catch(() => null)`, so an operational failure (timeout, 5xx, parse error) is no longer indistinguishable from a legitimately empty stat table; the IV reading still returns in every case, preserving partial success. ([#20](https://github.com/cyanheads/usgs-water-mcp-server/issues/20))
- **`water_get_conditions` upstream-error classification** — the handler's ad hoc `msg.includes('unavailable' | 'ServiceUnavailable' | 'timed out')` catch is replaced with `classifyNwisFailure()`, which branches on `err.code`. The substring checks never matched the service's real message `"NWIS returned HTTP 503: Service Unavailable"` (the capital "U" fails `'unavailable'`, the space fails `'ServiceUnavailable'`), so a genuine 5xx could slip past `upstream_error`. ([#18](https://github.com/cyanheads/usgs-water-mcp-server/issues/18))

## Dependencies
